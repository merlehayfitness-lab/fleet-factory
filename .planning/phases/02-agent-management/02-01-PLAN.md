    Migration to add `frozen` status to the agents table CHECK constraint. The current constraint only allows: provisioning, active, paused, error, retired.

    ```sql
    -- 011_agent_frozen_status.sql
    -- Add 'frozen' lifecycle status to agents table for emergency freeze control (AGNT-05)
    ALTER TABLE public.agents
      DROP CONSTRAINT IF EXISTS agents_status_check;

    ALTER TABLE public.agents
      ADD CONSTRAINT agents_status_check
      CHECK (status IN ('provisioning', 'active', 'paused', 'frozen', 'error', 'retired'));
    ```

    **packages/db/schema/012_agent_templates_rls.sql:**
    Create the `is_platform_admin()` helper function and INSERT/UPDATE/DELETE RLS policies for agent_templates. For MVP, a platform admin is any authenticated user who is an owner of at least one business.

    ```sql
    -- 012_agent_templates_rls.sql
    -- Platform admin helper and template write policies (AGNT-01 CRUD support)

    -- Platform admin check: any user who is an owner of at least one business
    CREATE OR REPLACE FUNCTION public.is_platform_admin()
    RETURNS boolean
    LANGUAGE sql
    SECURITY DEFINER
    SET search_path = ''
    AS $$
      SELECT EXISTS (
        SELECT 1 FROM public.business_users bu
        WHERE bu.user_id = (SELECT auth.uid())
          AND bu.role = 'owner'
      );
    $$;

    -- Template write policies
    CREATE POLICY "agent_templates_insert_admin"
      ON public.agent_templates FOR INSERT
      TO authenticated
      WITH CHECK (public.is_platform_admin());

    CREATE POLICY "agent_templates_update_admin"
      ON public.agent_templates FOR UPDATE
      TO authenticated
      USING (public.is_platform_admin())
      WITH CHECK (public.is_platform_admin());

    CREATE POLICY "agent_templates_delete_admin"
      ON public.agent_templates FOR DELETE
      TO authenticated
      USING (public.is_platform_admin());
    ```

    **packages/core/agent/lifecycle.ts:**
    Create the agent lifecycle state machine. Export:
    - `VALID_TRANSITIONS: Record<AgentStatus, AgentStatus[]>` -- the transition map:
      - provisioning -> [active, error]
      - active -> [paused, frozen, error, retired]
      - paused -> [active, frozen, retired]
      - frozen -> [active, retired]
      - error -> [active, retired]
      - retired -> [] (terminal state)
    - `canTransition(from: AgentStatus, to: AgentStatus): boolean`
    - `assertTransition(from: AgentStatus, to: AgentStatus): void` -- throws descriptive error if transition is invalid
    - `getValidTransitions(from: AgentStatus): AgentStatus[]` -- returns available next states

    **packages/core/agent/service.ts:**
    Agent service functions that Server Actions delegate to:

    `transitionAgentStatus(supabase, agentId, businessId, newStatus)`:
    1. Fetch current agent status from agents table (scoped by business_id)
    2. Call assertTransition(currentStatus, newStatus) -- throws if invalid
    3. Update agent status in the database
    4. Insert audit_log entry with action `agent.${newStatus}`, entity_type 'agent', entity_id agentId, metadata { previous_status, new_status }
    5. Throw descriptive errors on failures

    `updateAgentConfig(supabase, agentId, businessId, config)`:
    1. Fetch current agent (verify it exists and belongs to business)
    2. Update agent fields (system_prompt, tool_profile, model_profile) -- only update fields that are provided
    3. Insert audit_log entry with action 'agent.config_updated'
    4. Throw on failure
