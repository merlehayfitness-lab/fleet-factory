    **Create packages/core/slack/slack-blocks.ts:**
    Slack Block Kit message formatters for structured agent responses.

    - `formatAgentResponseBlocks(content: string, toolCalls?: ToolCallTrace[]): { blocks: object[], text: string }`:
      - text: plain text fallback of content (Slack requires text even when blocks are present)
      - blocks array with:
        - Section block with mrkdwn text for the main response content
        - If toolCalls exist: Divider block, then for each tool call: a Context block with "wrench" emoji + tool name + summary
      - Keep blocks simple and readable -- no overuse of formatting

    - `formatToolCallAttachment(trace: ToolCallTrace): object`:
      - Returns a Slack attachment object with:
        - color: "#36a64f" (green, indicating tool execution)
        - fields: [{ title: trace.toolName, value: trace.summary, short: true }]
        - fallback: `${trace.toolName}: ${trace.summary}`
      - Used when attaching tool call details as secondary content

    - `getAgentEmoji(departmentType: string): string`:
      - Returns an emoji string for the agent's icon_emoji in chat.postMessage
      - Mapping: sales -> ":chart_with_upwards_trend:", support -> ":headphones:", operations -> ":gear:", owner -> ":crown:", default -> ":robot_face:"
      - Per user decision: agents should be visually distinct with their own icons/emoji

    **Enhance packages/core/slack/slack-messages.ts:**
    Update the message handler and posting functions:

    - Update `postAgentResponseToSlack`:
      - Accept additional params: agentName (string), departmentType (string), toolCalls (ToolCallTrace[])
      - Use Block Kit formatting via formatAgentResponseBlocks for rich message display
      - Set username to agentName for per-agent identity
      - Set icon_emoji via getAgentEmoji(departmentType) for visual distinction
      - Pass blocks and text to chat.postMessage
      - If toolCalls exist, include attachments via formatToolCallAttachment
      - Return the message ts for storage

    - Update `handleInboundSlackMessage`:
      - Add @mention detection: check if event.text contains `<@{botUserId}>` (Slack's @mention format)
      - Per user decision: agents respond ONLY when @mentioned
      - If message does NOT contain @mention: store in Supabase as a user message (for display in admin panel) but do NOT route to agent
      - If message DOES contain @mention: store in Supabase AND route to agent via routeAndRespond, then post response back to Slack
      - Strip the @mention from the message text before routing to agent (clean input)
      - Look up bot_user_id from slack_installations table for the @mention check
      - When lead agent delegates to sub-agent: the sub-agent response posts in the SAME channel (not a sub-channel) per user decision -- this is handled naturally since routeAndRespond already picks the right agent and we post to the same channel

    - Add `getSlackFeedMessages(supabase, businessId, departmentId, limit?, before?): Promise<ChatMessage[]>`:
      - Fetches messages from Supabase messages table for a department, filtered to only show messages that have slack_ts (i.e., Slack-synced messages)
      - This is the data source for the "embedded Slack feed view" in the admin panel (SLACK-02)
      - Returns ChatMessage[] format compatible with existing message display components
      - Ordered by created_at ASC

    **Update packages/core/server.ts:**
    - Re-export formatAgentResponseBlocks, getAgentEmoji from slack/slack-blocks
    - Re-export getSlackFeedMessages from slack/slack-messages

      2. Get the agent's department type for emoji selection
      3. Call postAgentResponseToSlack with the agent response content, agent name, department type, and tool calls

    - This is a best-effort integration: wrap Slack posting in try/catch, log errors but don't fail the response
