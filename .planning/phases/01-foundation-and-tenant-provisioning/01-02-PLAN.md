Set up Supabase Auth with cookie-based sessions, create the sign-in page, and protect all dashboard routes with middleware + Server Component auth checks.

Purpose: Authentication is the gateway to everything. Without auth, RLS policies have no user context (auth.uid() is null). This plan establishes the complete auth flow so that subsequent plans can assume an authenticated user exists.

Output: Working sign-in/sign-up pages, Supabase SSR client utilities, middleware route protection, and a protected dashboard layout shell.
</objective>

  <files>

  <action>
    Install Supabase dependencies in apps/web: `pnpm --filter web add @supabase/supabase-js @supabase/ssr`

    **apps/web/_lib/env.ts:**
    Export helper functions that wrap environment variable access (per CLAUDE.md rule "Put environment variable access behind helper functions"):
    - `getSupabaseUrl()`: returns `process.env.NEXT_PUBLIC_SUPABASE_URL` with a throw if missing
    - `getSupabaseAnonKey()`: returns `process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY` with a throw if missing
    - `getSupabaseServiceRoleKey()`: returns `process.env.SUPABASE_SERVICE_ROLE_KEY` with a throw if missing (server-only, never exposed to client)

    **apps/web/_lib/supabase/server.ts:**
    Follow the exact Supabase SSR pattern from research. Export `createServerClient()` that:
    - Imports `createServerClient as createSSRClient` from `@supabase/ssr`
    - Gets cookieStore from `next/headers` cookies()
    - Creates client with getSupabaseUrl() and getSupabaseAnonKey()
    - Passes cookies.getAll() and cookies.setAll() handlers
    - Returns the Supabase client

    **apps/web/_lib/supabase/client.ts:**
    Export `createBrowserClient()` that:
    - Imports `createBrowserClient as createBrowserSupabaseClient` from `@supabase/ssr`
    - Uses the NEXT_PUBLIC_ env vars directly (client-side, available via Next.js)
    - Returns the browser Supabase client

    **apps/web/_lib/supabase/middleware.ts:**
    Follow the exact middleware pattern from research. Export `updateSession(request: NextRequest)` that:
    - Creates a Supabase server client with request/response cookie bridging
    - Calls `supabase.auth.getUser()` to refresh the session
    - If no user AND path is not /sign-in or /sign-up or /auth/callback, redirect to /sign-in
    - Returns the supabaseResponse with updated cookies

    **apps/web/app/auth/callback/route.ts:**
    Create a GET route handler for OAuth/email confirmation callbacks:
    - Extract `code` from searchParams
    - If code exists, exchange it with `supabase.auth.exchangeCodeForSession(code)`
    - Redirect to /businesses on success
    - Redirect to /sign-in on error

    Create `.env.local.example` in apps/web with placeholder values:
    ```
    NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
    NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
    SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
    ```
  </action>

    Verify env.ts throws on missing env vars (not silently returning undefined).
    Run `pnpm turbo typecheck` to confirm no type errors.
  </verify>
  <done>
    Supabase client utilities are created for server, client, and middleware contexts. Environment variables are accessed through helper functions. Auth callback route handles code exchange. All utilities typecheck.

  <action>
    Install UI dependencies: `pnpm --filter web add sonner` and initialize shadcn/ui: `pnpm dlx shadcn@latest init --defaults --yes` in apps/web directory. Then add required shadcn components: `pnpm dlx shadcn@latest add button card input label` in apps/web directory.

    **apps/web/middleware.ts:**
    - Import updateSession from './_lib/supabase/middleware'
    - Export middleware function that calls updateSession(request) and returns its result
    - Export config.matcher that matches all routes EXCEPT: _next/static, _next/image, favicon.ico, public assets (svg, png, jpg, etc.)

    **apps/web/app/(auth)/layout.tsx:**
    A centered layout for auth pages. Simple flex container, centered both axes, min-h-screen. No nav, no sidebar. Just the auth form centered on screen.

    **apps/web/app/(auth)/sign-in/page.tsx:**
    A Client Component ("use client") sign-in page with:
    - Card component containing the form
    - Title "Sign in to Agency Factory"
    - Email input (type="email", required)
    - Password input (type="password", required)
    - Submit button "Sign in"
    - Link to /sign-up for new users
    - Uses createBrowserClient() to call supabase.auth.signInWithPassword({ email, password })
    - On success: router.push('/businesses') and router.refresh()
    - On error: display error message below the form
    - Loading state on the button while signing in
    - Uses shadcn/ui Card, Input, Button, Label components

    **apps/web/app/(auth)/sign-up/page.tsx:**
    Similar to sign-in but:
    - Title "Create your account"
    - Calls supabase.auth.signUp({ email, password })
    - On success: show "Check your email to confirm your account" message
    - Link to /sign-in for existing users

    **apps/web/app/(dashboard)/layout.tsx:**
    A Server Component that:
    - Calls createServerClient() and then supabase.auth.getUser() (defense in depth -- CVE-2025-29927)
    - If no user, redirect('/sign-in')
    - Renders a minimal shell: sidebar placeholder (just a div with "Agency Factory" text and a sign-out button) + main content area with {children}
    - The sign-out button triggers a Server Action that calls supabase.auth.signOut() and redirects to /sign-in
    - This layout wraps all /businesses/* routes

    IMPORTANT: The sign-out Server Action must be in a separate file (e.g., apps/web/_actions/auth-actions.ts) because it uses 'use server' directive. It should call createServerClient(), then supabase.auth.signOut(), then redirect('/sign-in').

    IMPORTANT: Use getUser() NOT getSession() for Server Component auth checks. getUser() validates with the Supabase Auth server; getSession() only reads the local JWT which could be tampered with.
  </action>
