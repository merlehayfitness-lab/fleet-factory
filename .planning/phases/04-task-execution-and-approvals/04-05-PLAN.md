The core fix is in `packages/core/worker/tool-runner.ts`. The `runAgentTask` function currently calls `getToolRiskLevel(toolName)` to get the catalog-based risk level and uses that directly for approval gating. We need to also call `evaluateRisk` from the policy engine and take the higher of the two risk levels.

**Step 1: Add import for evaluateRisk**

Add to the existing imports at the top of tool-runner.ts:
```typescript
import { evaluateRisk } from "../approval/policy-engine";
```

**Step 2: Add a helper function to compare risk levels**

Add a helper function after the imports and interfaces (before the `runTool` function):
```typescript
const RISK_ORDER: Record<string, number> = { low: 0, medium: 1, high: 2 };

/**
 * Return the higher of two risk levels.
 * Used to combine catalog-based risk with policy-engine risk.
 */
function maxRiskLevel(
  a: "low" | "medium" | "high",
  b: "low" | "medium" | "high",
): "low" | "medium" | "high" {
  return (RISK_ORDER[a] ?? 0) >= (RISK_ORDER[b] ?? 0) ? a : b;
}
```

**Step 3: Update the risk evaluation in `runAgentTask`**

In the `runAgentTask` function, the loop at step 3 (around line 168) currently does:
```typescript
for (const toolName of requestedTools) {
    const riskLevel = getToolRiskLevel(toolName);
```

Replace that risk determination with a dual evaluation that queries the policy engine and takes the higher level:
```typescript
for (const toolName of requestedTools) {
    const catalogRisk = getToolRiskLevel(toolName);

    // Consult database-backed policy engine for this tool's action pattern
    let policyRisk: "low" | "medium" | "high" = catalogRisk;
    try {
      const policyResult = await evaluateRisk(supabase, toolName);
      policyRisk = policyResult.riskLevel;
    } catch {
      // Policy engine failure: fall back to catalog risk only (fail-open to existing behavior)
    }

    // Take the higher of catalog risk and policy engine risk
    const riskLevel = maxRiskLevel(catalogRisk, policyRisk);
```

The rest of the for-loop body remains unchanged -- it already uses `riskLevel` for the approval gate checks (high -> needs approval, medium + not trusted -> needs approval, low -> auto-execute).

**Important notes:**
- `runAgentTask` is already `async`, so the `await evaluateRisk(...)` call works naturally
- The `supabase` client is already a parameter to `runAgentTask` -- no new parameters needed
- On policy engine failure (database error, network issue), we catch the error and fall back to catalog risk only. This preserves existing behavior and prevents the policy engine from breaking task execution
- The `runTool` function (single tool execution) is NOT changed -- it deliberately does NOT check approvals (comment in code: "approval gate check happens in the executor before calling runTool"). Risk evaluation for approval gating happens only in `runAgentTask`
- The approval action string in the needs_approval return now also includes which source drove the risk level, for transparency in the approval UI

**Step 4: Enhance the approval action strings**

Update the two `needsApproval` return blocks to include which risk source was decisive. In the high-risk block:
```typescript
approvalAction: `Execute high-risk tool: ${toolName}${policyRisk === "high" && catalogRisk !== "high" ? " (elevated by policy)" : ""}`,
```

In the medium-risk block:
```typescript
approvalAction: `Execute medium-risk tool: ${toolName} (agent not trusted)${policyRisk === "medium" && catalogRisk !== "medium" ? " (elevated by policy)" : ""}`,
```

This way, if the policy engine elevated a tool's risk above the catalog level, the admin can see that in the approval action summary.
