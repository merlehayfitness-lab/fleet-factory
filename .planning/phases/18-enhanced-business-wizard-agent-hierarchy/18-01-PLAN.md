  <action>
    **Extend the existing `createBusinessV2` server action** to create V2 departments and agents from the selected template IDs:

    The current `createBusinessV2` calls `provisionBusinessTenant` (base RPC creating 4 default departments) and then saves subdomain, API keys, and allocates ports. The gap is that `selectedTemplateIds` is parsed from FormData but not used to create the correct departments/agents.

    **After the base RPC call, add template-aware provisioning:**

    1. **Parse selected template IDs** from FormData (already done: `formData.get("selectedTemplateIds")` as JSON array of strings).

    2. **Fetch V2 templates** from `agent_templates` table where `id IN (selectedTemplateIds)`. These have `role_level`, `reporting_chain`, `token_budget`, `parent_template_id`, `department_type`, `skills_package`, `mcp_servers`.

    3. **Determine departments to create:** Group selected templates by `department_type`. For each unique department_type not already created by the base RPC (owner, sales, support, operations are created by default), INSERT a new department: marketing, rd, executive, hr as needed.

    4. **Create agents from selected templates in hierarchy order:**
       - Sort templates by `role_level` ascending (0=CEO first, then 1=dept heads, then 2=specialists)
       - For each template, create an agent record:
         - `business_id`, `department_id` (looked up from departments), `name` from template, `status = 'active'`
         - `system_prompt`, `tool_profile`, `model_profile` from template
         - `token_budget` from template
         - `parent_agent_id`: resolve from the agent created for `parent_template_id` (maintain a `Map<templateId, agentId>` during creation)
       - Use Supabase `.insert()` per role_level batch, collecting IDs for parent resolution

    5. **Remove or skip duplicate agents:** The base RPC already creates 4 default agents (one per default department). If those same templates are in the selected list, skip re-creation. Check by matching `template_id` on the agent's record.

    6. **Update redirect target:** Change `redirect(/businesses/${businessId})` to `redirect(/businesses/${businessId}/deployments)` per user decision (on submit, redirect to deployment status page).

    **Important implementation details:**
    - The base RPC creates business + membership + 4 departments + 4 agents + deployment job atomically. V2 agent creation is additive (not replacing the RPC). If a V2 template maps to an already-created department, reuse that department.
    - Keep the SSH deploy conditional branch but pass real agent IDs (not template IDs) to `sshDeployBusiness`.
    - If template-aware provisioning fails, the business still exists (from RPC). Return the error but don't orphan the business -- it can be fixed from settings.
    - Use `as any` sparingly; type the template query result properly.

    **Patterns:**
    - Thin actions: the logic for resolving templates to agents could be extracted to a helper function within the same file or in packages/core, but for this phase keep it in the action to move fast.
    - redirect() must be called OUTSIDE try/catch (NEXT_REDIRECT throws internally).
