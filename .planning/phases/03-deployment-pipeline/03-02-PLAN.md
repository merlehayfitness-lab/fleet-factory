<objective>
Build the deployment job queue with status tracking state machine, deploy/retry/rollback service functions, config snapshot versioning, and Server Actions.

Purpose: This plan creates the deployment execution engine that drives the deployment pipeline. The state machine (DEPL-07) validates all status transitions. The deploy service (DEPL-02) orchestrates the build pipeline: fetch data, run generators, store snapshots. Retry (DEPL-08) re-runs from current state. Rollback (DEPL-09) restores from a previous snapshot. Config snapshots (DEPL-10) version all agent configs per deployment. Server Actions wire everything to the UI layer built in 03-03.

Output: Deployment lifecycle state machine, deployment service with triggerDeployment/retryDeployment/rollbackDeployment, config snapshot create/restore, and Server Actions for the UI.
</objective>

    **packages/core/deployment/lifecycle.ts:**
    Deployment state machine following the same pattern as agent/lifecycle.ts:

    ```typescript
    import type { DeploymentStatus } from "../types/index";

    export const DEPLOYMENT_TRANSITIONS: Record<DeploymentStatus, DeploymentStatus[]> = {
      queued: ["building"],
      building: ["deploying", "failed"],
      deploying: ["live", "failed"],
      live: ["rolled_back"],
      failed: ["queued"],  // retry creates new record, but failed can transition to queued conceptually
      rolled_back: [],     // terminal for this deployment record
    };
    ```

    Export:
    - `canTransitionDeployment(from: DeploymentStatus, to: DeploymentStatus): boolean`
    - `assertDeploymentTransition(from: DeploymentStatus, to: DeploymentStatus): void` -- throws if invalid
    - `getValidDeploymentTransitions(from: DeploymentStatus): DeploymentStatus[]`

    **packages/core/deployment/snapshot.ts:**
    Config snapshot creation and restoration:

    ```typescript
    interface SnapshotAgent {
      id: string;
      name: string;
      system_prompt: string;
      tool_profile: Record<string, unknown>;
      model_profile: Record<string, unknown>;
      department_id: string;
      status: string;
    }

    interface ConfigSnapshot {
      version: number;
      business: { id: string; name: string; slug: string };
      agents: SnapshotAgent[];
      departments: Array<{ id: string; name: string; type: string }>;
      integrations: Array<{ id: string; agent_id: string | null; type: string; provider: string; status: string }>;
      generated_at: string;
    }
    ```

    Export:
    - `createConfigSnapshot(version, business, agents, departments, integrations): ConfigSnapshot` -- assembles the snapshot object from the provided data
    - `restoreFromSnapshot(snapshot: ConfigSnapshot): { agents: SnapshotAgent[]; departments: ...; integrations: ... }` -- extracts the restorable data from a snapshot (used by rollback to know what state to restore)
    - Export the `ConfigSnapshot` type

    **packages/core/deployment/service.ts:**
    The main deployment service with three core operations:

    `triggerDeployment(supabase, businessId, triggeredBy)`:
    1. Fetch the business record (verify exists, check status is not 'disabled')
    2. Fetch the latest deployment for this business (to get last version number)
    3. Calculate next version: lastVersion + 1 (or 1 if first deployment)
    4. Fetch all agents for this business (with department joins)
    5. Fetch all integrations for this business
    6. Create config snapshot using createConfigSnapshot()
    7. Insert new deployment record with status 'queued', version, triggered_by, config_snapshot
    8. Transition to 'building' -- update status
    9. Generate all artifacts using packages/runtime generators:
       - Import generateTenantConfig, generateDockerCompose, generateEnvFile, generateAllAgentConfigs from @agency-factory/runtime
       - Fetch and decrypt secrets for the business (import decrypt from crypto/encryption)
       - Call each generator with the appropriate data
       - Store generated artifacts in the config_snapshot jsonb field under an "artifacts" key:
         ```json
         {
           "version": 2,
           "business": {...},
           "agents": [...],
           "artifacts": {
             "tenant_config": "...",
             "docker_compose": "...",
             "env_file": "...",
             "agent_configs": [{ "agent_id": "...", "filename": "...", "content": "..." }]
           }
         }
         ```
    10. Transition to 'deploying' -- update status and started_at
    11. For MVP: no actual Docker deployment. Just transition to 'live' with completed_at.
    12. Update all agents in the business to 'active' status (if currently 'provisioning')
    13. Update the business status to 'active' (if currently 'provisioning')
    14. Insert audit_log entry with action 'deployment.completed', metadata: { version, agent_count }
    15. On ANY error at any stage: catch, transition to 'failed' with error_message, insert audit_log with action 'deployment.failed'
    16. Return the deployment record

    IMPORTANT: Wrap the entire pipeline in a try/catch. On failure, update deployment to 'failed' with the error message. Never leave a deployment stuck in 'building' or 'deploying'. The decrypt function needs the ENCRYPTION_KEY env var -- if it's not set in development, handle gracefully by using empty secrets array and logging a warning.

    `retryDeployment(supabase, businessId, failedDeploymentId, triggeredBy)`:
    1. Fetch the failed deployment (verify status is 'failed')
    2. Call triggerDeployment() to create a fresh deployment from current state
    3. Return the new deployment

    `rollbackDeployment(supabase, businessId, targetVersion, triggeredBy)`:
    1. Fetch the target deployment by business_id and version (verify it was 'live' or 'rolled_back')
    2. Extract config_snapshot from the target deployment
    3. Create a new deployment record with next version, status 'queued', and the old config_snapshot
    4. Transition through building -> deploying -> live
    5. Mark the currently-live deployment as 'rolled_back'
    6. Insert audit_log with action 'deployment.rolled_back', metadata: { from_version, to_version }
    7. Return the new deployment

    `getDeploymentHistory(supabase, businessId, limit = 20)`:
    1. Fetch deployments ordered by created_at DESC with limit
    2. Return array of deployment records

    IMPORTANT: Never wrap redirect() in try/catch. Follow the same thin Server Action pattern as agent-actions.ts -- service functions do the real work, Server Actions are thin wrappers.

    **apps/web/_lib/env.ts:**
    Add a new helper for the encryption key:

    ```typescript
    /**
     * Server-only. Used for encrypting/decrypting tenant secrets.
     * Optional in development -- returns empty string if not set to allow
     * deployment pipeline to run without real encryption.
     */
    export function getEncryptionKey(): string {
      const key = process.env.ENCRYPTION_KEY;
      if (!key) {
        console.warn(
          "ENCRYPTION_KEY not set. Secrets encryption will not work. " +
          "Set a 64-character hex string in .env.local for production use."
        );
        return "";
      }
      return key;
    }
    ```

    **apps/web/_actions/deployment-actions.ts:**
    "use server" module with thin Server Actions for deployment operations:

    `deployAction(businessId: string)`:
    - createServerClient(), getUser(), redirect if no user
    - Call triggerDeployment(supabase, businessId, user.id) from @agency-factory/core
    - revalidatePath for /businesses/${businessId}/deployments and /businesses/${businessId}
    - Return { success: true, deploymentId } or { error: string }
    - IMPORTANT: Never wrap redirect() in try/catch

    `retryDeploymentAction(businessId: string, deploymentId: string)`:
    - Same auth pattern
    - Call retryDeployment(supabase, businessId, deploymentId, user.id)
    - revalidatePath for deployments page
    - Return { success: true, deploymentId } or { error }

    `rollbackDeploymentAction(businessId: string, targetVersion: number)`:
    - Same auth pattern
    - Call rollbackDeployment(supabase, businessId, targetVersion, user.id)
    - revalidatePath for deployments page
    - Return { success: true, deploymentId } or { error }

    `getDeploymentsAction(businessId: string)`:
    - Same auth pattern
    - Call getDeploymentHistory(supabase, businessId)
    - Return { deployments } or { error }

    Each action follows the thin Server Action pattern: under 20 lines, delegates to the core service, handles errors with try/catch (but redirect outside try/catch), returns simple result objects.
